# Iteration 13 Implementation Prompt — codex_context_knowledge_retrieval

Implement the **knowledge retrieval engine** for artifacts generated by Iterations 11 and 12.

## Goal

Given a user request, identify the most relevant mod knowledge and load the **smallest useful set of derived artifacts** from `.codex_library/` instead of loading raw documents or overly broad context.

This iteration is the retrieval layer that turns the local knowledge library into a low-cost, reusable context system.

---

## 1. Detect candidate mods

For an incoming request, infer one or more likely candidate mods.

Sources for inference may include:
- explicit user phrasing
- previously active mod context
- simple keyword / alias mapping
- available mod registry entries
- task-specific memory if present

Example:
- request mentions onboarding, usability, interface, visual clarity -> candidate mod `ux`
- request mentions accessibility, screen reader, contrast, keyboard navigation -> candidate mod `accessibility`

Keep detection generic and extensible.

---

## 2. Detect candidate topics

Infer likely topics / subtopics from the request.

Use available lightweight signals:
- request keywords
- topic indices from Iteration 12
- retrieval maps
- note / summary filenames
- manifest metadata if useful

Example:
- request: "improve onboarding clarity"
- candidate topic: `onboarding`

---

## 3. Load lightweight indices first

Before loading any note or summary bodies, prefer to read only small routing artifacts such as:

```text
.codex_library/mods/<mod_id>/indices/
```

Especially:
- topic index
- keyword index
- retrieval map

Goal:
decide what to load next using the cheapest possible artifacts first.

---

## 4. Retrieve the smallest useful set of artifacts

Based on candidate mod(s) and topic(s), select a compact set of artifacts such as:

- one or more topical notes
- one summary
- one retrieval hint file
- minimal manifest metadata if needed

Prefer:
- `notes/`
- `summaries/`
- `indices/`

Avoid:
- raw source files in `inbox/`
- loading everything under a mod
- overly broad context packs

---

## 5. Budget-aware retrieval

Integrate with the existing engine philosophy of context cost control.

Retrieval should:
- select fewer artifacts when the task is narrow
- expand only if confidence is low
- prefer short notes before long summaries
- remain compatible with the broader context optimizer

If the repo already has a budget or optimizer concept, route retrieval through that pattern instead of bypassing it.

---

## 6. Output a retrieval pack

Produce a compact structured retrieval result that the rest of the engine can consume.

Suggested shape:

```json
{
  "mods": ["ux"],
  "topics": ["onboarding"],
  "selected_artifacts": [
    ".codex_library/mods/ux/notes/onboarding.md",
    ".codex_library/mods/ux/summaries/practical_rules.md"
  ],
  "strategy": "topic_first_minimal_pack"
}
```

Exact format may vary, but it should be easy to inspect and debug.

---

## 7. Fall-back behavior

If:
- no relevant mod exists
- no indices exist yet
- no matching artifact is found

then degrade gracefully.

Possible fallbacks:
- report no learned corpus for that domain
- fall back to smaller generic mod summaries if present
- suggest the user ingest documents for that mod
- return an empty retrieval pack instead of failing

Do not over-load unrelated artifacts just to avoid returning empty.

---

## 8. MCP integration

Be ready to use these MCPs when available:

- **filesystem MCP**
  - load indices, notes, summaries, manifests
- **git MCP**
  - optional repo-aware hints about recently touched domains/files
- **fetch MCP**
  - optional enrichment only when explicitly needed
- **playwright MCP**
  - optional live UI context for relevant requests

Important:
- all MCP integrations remain optional
- the retrieval engine must still work in local-only mode
- do not add GitHub MCP dependency

## 10. Acceptance criteria

Implementation is correct if all of the following are true:

- incoming requests can be mapped to one or more candidate mods
- retrieval starts from small indices rather than raw documents
- only a minimal useful set of artifacts is selected
- selected artifacts prefer notes / summaries over source docs
- retrieval is budget-aware and optimizer-friendly
- missing knowledge degrades gracefully
- MCP support is present but optional

---

## 11. Output quality bar

Prefer transparent, debuggable retrieval over opaque cleverness.

The retrieval engine should be easy to inspect:
- why a mod was chosen
- why a topic was chosen
- why a file was selected

Optimize for reusable low-cost context selection.
